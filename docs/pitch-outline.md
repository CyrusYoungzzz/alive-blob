## Slide 2 — 问题与洞察（45s）

**画面**：社会压力数据/年轻人疲惫的画面 → 过渡到 Blob 被捏/拍打的画面

**讲稿要点**：

### 洞察 1：压力没有出口

> "我们观察到一个普遍的问题 — 现代人有太多压力，但没有足够的发泄空间。"

- 你不能对人发火，不能摔东西，不能对着空气大喊
- 人们需要一种**简单、有趣、即时反馈**的方式来泄压 — 而且不会伤害任何人
- 关键词：**快速反馈** — 你打它一下，它立刻有表情反应、步态变化，这个反馈回路本身就是解压

> "我们想做的，就是给你一个可以安全发泄的小东西 — 你捏它、拍它、戳它，它不会生气，反而会给你一个滑稽的表情。"

### 洞察 2：软体 = 生命感 + 滑稽感

> "为什么是软体？因为硬邦邦的机器人捏着不舒服，也不好笑。"

- 软体机器人天生更有**生命感** — 硅胶的柔软和形变让人联想到活的生物
- 同时自带**滑稽感** — 一个软乎乎的东西笨拙地走路，本身就让人想笑
- 这种「又蠢又可爱」的气质，正是泄压的最佳载体 — 你不会���一个可爱的东西产生负罪感

> "Alive Blob — 一个你可以随便揉捏、怎么打都不会坏、还会用表情回应你的软体小生物。"

## Slide 5 — 创新性：技术深度（60s）

**画面**：三栏架构图 — Blob Engine / Eye App / Web Server

**讲稿要点**（挑 3 个最硬的讲）：

### 1. 气动软体行走 — 「为什么不用电机？」
> "传统机器人用舵机和齿轮，我们用空气。一个气泵 + 两个电磁阀，通过时序控制让硅胶双腿交替膨胀收缩。"

- 零齿轮、零轴承 — 没有磨损件
- 4 种步态模式映射到情绪状态（平稳/暂停/不对称/跺脚）
- 三层安全机制：30s 泵看门狗 → 启动前泄压 → 进程崩溃 GPIO 归零

### 2. 触摸→情绪→表情→步态 全链路 <100ms
> "从手指按下到表情变化和步态切换，延迟不到 100 毫秒。"

- MPR121 12 通道电容触摸 → 区分头部/身体/腿部
- 7 种情绪状态机 + 触摸模式识别（单击/连击/长按/多点触控）
- 双层视频交叉淡入 + Canvas 粒子叠加层，实现无缝表情切换

### 3. AIGC 换脸 — 「解铃还须系铃人」

> "每个人的情绪，都可能被某几个特定的人牵动 — 老板、甲方、前任、室友……"
> "既然是他让你不爽的，那就让他来承受你的怒火。"

- 上传「那个人」的照片 → AI 自动生成 7 种表情动画
- 现在这个 Blob 顶着他的脸，在你桌上笨拙地走路、被你随便揉捏
- **你打它，它用那张脸露出委屈/害怕/生气的表情** — 泄压闭环达成
- 本质：把抽象的情绪压力，绑定到一个**具体的、可交互的、无害的**物理对象上

> "解铃还须系铃人 — 只不过这次，系铃的人变成了一个你可以随便捏的软体小东西。"

**技术实现**：字节即梦 AIGC API → 人脸检测 + 7 情绪视频批量生成 → 角色热插拔

## 附：Q&A 预案

| 可能的问题 | 建议回答 |
|-----------|---------|
| 为什么用气动不用电机？ | 软体 = 柔软触感 + 无磨损 + 成本低；电机做不到「捏着舒服」这件事 |
| 和 Vector/Cozmo 的区别？ | 它们是刚性轮式机器人；我们是软体步行 + 触感 + 可换脸，物理交互维度不同 |
| 安全性怎么保证？ | 三层看门狗（泵时间/泄压/GPIO 归零），即使软件崩溃硬件也安全 |
| 商业化最大挑战？ | 量产模具和供应链；但核心器件都是标准工业品，无定制 ASIC |
| AI 部分具体做了什么？ | AIGC 表情生成（即梦 API），未来计划加入语音交互和情绪学习 |
| 能量续航？ | 目前外接 12V 适配器；消费版计划用 3S 锂电池，预估 2-3 小时续航 |

## 演讲技巧提醒

1. **视频 Demo 是核心武器** — 让评委「看到」而不是「听到」，视频播放时少说话
2. **技术深度选择性展开** — 5 分钟不要贪多，只讲气动 + 全链路延迟 + AIGC 三个点
3. **数字要脱口而出** — ¥350、<100ms、48 小时、7 种情绪、4 种步态
4. **结尾要有记忆点** — 最后一句话是评委记住你的关键
5. **准备一个 30 秒版本** — 万一被要求缩短，只讲：软体行走 + 触摸表情 + ¥350 + 48 小时
