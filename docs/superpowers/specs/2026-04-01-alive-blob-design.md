# Alive Blob - 设计文档

一个会走路、能感知触摸、能用 AIGC 生成的人脸表情回应你的抽象软体机器人。为 48 小时黑客松打造。

## 项目目标

做一个让人觉得「它是活的」的物理陪伴体。用户上传任何人的照片，用 AIGC 工具生成表情视频，Blob 会在头顶小圆屏上展示这些表情，同时两条硅胶软腿交替充放气驱动它走路，不同情绪对应不同的步态节奏。摸它会触发情绪变化——比如「狂扁老板」场景：上传老板照片，拍它一下，它被打哭了还一瘸一拐地跑。

## 团队

- 3 人：1 名结构工程师（气动硅胶专家）+ 2 名软件工程师
- 黑客松要求实物硬件展示，评委上手体验

### GPIO 引脚分配

```
GPIO18 = 气泵开关 (HIGH = 开)
GPIO23 = 电磁阀1 - 左腿 (HIGH = 开, 常闭)
GPIO24 = 电磁阀2 - 右腿 (HIGH = 开, 常闭)
I2C SDA (GPIO2) = MPR121 触摸传感器
I2C SCL (GPIO3) = MPR121 触摸传感器
MPR121 I2C 地址 = 0x5A
```

### 气路拓扑

```
气泵 → 总气管 ─┬─→ 阀门1 (常闭) → 左腿气腔
               └─→ 阀门2 (常闭) → 右腿气腔

常闭 (NC) = 默认气路断开，通电打开进气
```

**走路周期：** 气泵持续运行，两个阀门交替开关
1. 阀门1 开 + 阀门2 关 → 左腿充气弯曲，身体向右倾
2. 阀门1 关 + 阀门2 开 → 右腿充气弯曲，身体向左倾
3. 交替重复 → 走路/摇摆动作

**放气方式：** 不需要���门的排气阀。阀门关闭后，硅胶腿自身弹性会把空气挤回总气管（此时另一条腿的阀门是开的，空气自然流向那边）。如果排气太慢，可在腿上留一个小排气孔辅助。

**气泵安全看门狗：**
- 气泵最长连续运行时间：30 秒。超时自动切断，需冷却 2 秒才能再次启动。
- Engine 进程退出或崩溃时：所有 GPIO 重置为 LOW（气泵停止、阀门关闭 → 停止走动）。通过 `atexit` 和 `signal` 处理器实现。

### 物理形态

参考 2 路气动 soft robot walker 设计：
- **刚性框架**：3D 打印或硬纸板搭建的上部框架，承载屏幕和布线
- **2.1 寸圆形 HDMI 屏**：嵌在框架顶部作为「头/脸」，约 30-40g
- **两条硅胶软腿**：底部两个独立气腔，交替充放气产生左右摇摆走路动作
- **Raspberry Pi**：不放在 walker 身上，通过气管和线缆从旁边连接（减重）
- **重心控制**：屏幕尽量低装，框架用轻质材料，确保软腿能撑住并稳定走路
- 电容触摸传感器贴在框架/硅胶表面，用户可以拍打、抚摸

## 情绪系统

### 7 种情绪

Blob 有 7 种情绪状态，每种情绪同时影响两个输出：屏幕表情视频和走路步态。设计基调是夸张搞怪——Blob 是一个戏精。

| 情绪 | 屏幕表情（AIGC 视频） | 步态节奏 |
|------|---------------------|---------|
| **Calm** 平静 | 放松的脸，慢慢眨眼，微微点头 | 慢而稳的散步（2秒/步） |
| **Happy** 开心 | 大笑，夸张咧嘴，眉毛上扬 | 快而轻的蹦跳（0.5秒/步） |
| **Curious** 好奇 | 歪头，挑眉，眼睛左右看 | 不对称节奏：快-慢-快，像东张西望 |
| **Sleepy** 困倦 | 打哈欠，眼睛下垂，头慢慢歪下去 | 极慢，走两步停一停，快睡着了 |
| **Excited** 兴奋 | 张大嘴，瞪大眼，头快速左右转 | 疯狂原地蹬腿，极快交替 |
| **Shy** 害羞 | 低头，脸红，偷偷抬眼看 | 小碎步，幅度很小（低充气量） |
| **Grumpy** 不爽 | 被打哭了，委屈皱眉，嘴角下撇 | 重踏步跺脚，充气猛烈 |

### 状态机

```
                    ┌──────────────┐
       超时无交互    │              │ 任意触摸
       (>60秒)     │   Sleepy     ├──────────┐
        ┌─────────►│              │          │
        │          └──────────────┘          ▼
        │                               ┌─────────┐
   ┌────┴─────┐                         │  Calm   │◄── 初始状态
   │          │◄────────────────────────│         │
   │  Calm    │                         └────┬────┘
   │          │──── 轻触 ────────► Happy      │
   └────┬─────┘                              │
        │         连续快速拍击                  │
        ├────────────────────► Excited        │
        │                                     │
        │         触摸新位置                    │
        ├────────────────────► Curious        │
        │                                     │
        │         持续按压同一点 (>2秒)          │
        ├────────────────────► Shy            │
        │                                     │
        │         拍打 (多点同时+高频)           │
        └────────────────────► Grumpy         │
                                              │
        所有非 Calm 情绪 ── 自然衰减 ──────────►┘
                           (10-30秒无交互)
```

关键设计决策：
- **自然衰减**：所有情绪在 10-30 秒无交互后渐变回 Calm，不会卡在某个情绪
- **触摸模式识别**：Engine 分析触摸频率、持续时间、触发通道数来判断手势类型
- **平滑过渡**：情绪切换时步态节奏渐变（逐渐加速/减速），视频交叉淡入淡出
- **手机覆盖**：手机控制面板可以强制切换到任意情绪，覆盖自动状态机

### 触摸模式识别

MPR121 提供 12 个离散触摸通道（0-11），不是连续的 XY 坐标。通道按物理区域映射到 Blob 身上：

```
MPR121 通道映射：
  通道 0-3:  头部/屏幕周围（框架顶部）
  通道 4-7:  身体/框架两侧
  通道 8-9:  左腿
  通道 10-11: 右腿
```

人员 A 按此区域图放置传感器，人员 B 读取通道号并按区域分类。

| 模式 | 检测规则 | 触发情绪 |
|------|---------|---------|
| 轻触 | 单通道触发，< 500ms | Happy |
| 持续按压 | 同一通道保持 > 2 秒 | Shy |
| 连续拍击 | 1 秒内 3 次以上触发 | Excited |
| 拍打 | 3 个以上通道同时触发 | Grumpy |
| 新区域触摸 | 触发通道与上次不在同一区域 | Curious |
| 无触摸 | 超过 60 秒无任何触摸 | Sleepy |

## AIGC 人脸系统

### 概述

Blob 的屏幕播放 AIGC 生成的人脸表情视频。用户用任意 AIGC 工具（可灵、Runway、ComfyUI、剪映等）在外部生成表情视频，打包上传为「角色包」。

### 视频规格

- 分辨率：480x480（匹配 2.1 寸圆屏）
- 时长：3-5 秒，可无缝循环
- 格式：MP4 (H.264)，Pi 硬件解码无压力
- 每个角色 7 个视频（每种情绪一个），总共约 20-40MB
- 支持多个角色包（川普包、马斯克包、老板包...），手机端切换

### 角色包目录结构

```
characters/
├── trump/
│   ├── manifest.json
│   ├── calm.mp4
│   ├── happy.mp4
│   ├── excited.mp4
│   ├── curious.mp4
│   ├── sleepy.mp4
│   ├── shy.mp4
│   └── grumpy.mp4
├── musk/
│   └── ...
└── boss/
    └── ...
```

### AIGC 视频生成指引（给内容制作者）

用目标人物照片生成每种情绪的 3-5 秒循环短视频：

| 情绪 | 生成方向 |
|------|---------|
| Calm | 表情放松，缓慢眨眼，微微点头 |
| Happy | 大笑，夸张咧嘴笑，眉毛弹跳 |
| Excited | 张大嘴，瞪圆眼，头疯狂摇晃 |
| Curious | 歪头，挑眉，眼睛东张西望 |
| Sleepy | 大哈欠，眼皮下垂，头慢慢歪下去 |
| Shy | 低头，脸红，偷偷抬眼看 |
| Grumpy | 哭脸，皱眉，嘴角下撇 |

## 软件模块

### 1. Blob Engine（Python — 大脑）

运行在 Pi 上的核心进程，协调所有模块。

```
blob-engine/
├── engine.py          # 主循环，模块协调
├── emotion_sm.py      # 情绪状态机
├── touch_handler.py   # 触摸模式识别
├── gait_controller.py # 步态控制器（替代原 pump_controller）
├── config.py          # 所有可调参数
└── requirements.txt
```

**步态控制器核心逻辑：**
```python
# 每种情绪对应的步态参数
GAIT_PARAMS = {
    "calm":    {"step_ms": 2000, "duty": 0.5, "pattern": "steady"},
    "happy":   {"step_ms": 500,  "duty": 0.5, "pattern": "steady"},
    "excited": {"step_ms": 200,  "duty": 0.5, "pattern": "steady"},
    "sleepy":  {"step_ms": 3000, "duty": 0.3, "pattern": "pause"},  # 走两步停一下
    "curious": {"step_ms": 800,  "duty": 0.6, "pattern": "asymmetric"},  # 快-慢-快
    "shy":     {"step_ms": 1500, "duty": 0.3, "pattern": "steady"},  # 小碎步
    "grumpy":  {"step_ms": 600,  "duty": 0.8, "pattern": "stomp"},   # 重踏步
}
# step_ms = 一步的时间（左→右为一步）
# duty = 充气占空比（0.8 = 充气猛烈，0.3 = 轻柔）
# pattern = 步态模式
```

**主循环（约 50ms 一轮）：**
```
读取触摸 → 判断手势类型 → 状态机决策 → 更新情绪 →
  ├→ 步态指令（GPIO：气泵 + 阀门交替）
  ├→ 切换表情视频（WebSocket → Eye App）
  └→ 状态同步（WebSocket → 手机）
```

### 2. Eye App（前端 — 脸）

Pi 上全屏 Chromium 打开的网页，基于视频播放。

```
eye-app/
├── index.html
├── style.css
├── player.js          # 双层视频切换引擎
├── overlay.js         # Canvas 触摸即时反馈层
├── transitions.js     # 情绪视频交叉淡入淡出
└── ws.js              # WebSocket 客户端
```

**视频播放设计：**
- 两个 `<video>` 元素叠在一起（A 层 / B 层）
- 当前情绪视频在 A 层循环播放
- 情绪切换时：B 层加载新视频 → 交叉淡入淡出（0.8秒）→ A/B 角色互换
- 视频设置 `loop` + `muted`，全屏铺满圆形屏幕
- 屏幕背景色跟随情绪变化（比如 Grumpy 时边缘泛红光）

**触摸即时反馈叠加层（Canvas 覆盖在视频上方）：**
- 轻触 → 触摸位置冒出波纹/心形粒子
- 拍打 → 整个屏幕抖动效果（CSS shake）
- 持续按压 → 按压区域出现涟漪扩散
- 作用：每次触摸都有「它感受到了」的即时感，不用等视频切换

## 通信协议

所有实时通信使用 WebSocket + JSON 消息，每条消息都有 `type` 字段做路由。

### WebSocket 架构

```
                  ┌───────── ws://localhost:8000/ws/eye ──────────┐
                  │                                                │
             ┌────▼─────┐                                   ┌─────┴──────┐
             │ Eye App  │                                   │   Blob     │
             │ (浏览器)  │                                   │  Engine    │
             └──────────┘                                   │  (Python)  │
                                                            └─────┬──────┘
             ┌──────────┐                                         │
             │ 手机      │── ws://192.168.x.x:8000/ws/mobile ─────┘
             │ (浏览器)  │
             └──────────┘
```

**Blob Engine 自己运行 WebSocket 服务器**（用 `websockets` 库），端口 8000，两个端点：
- `/ws/eye` — Eye App 连接到这里，只允许一个连接
- `/ws/mobile` — 手机客户端连接到这里，允许多个连接（都收到相同的状态同步消息）

**FastAPI 服务器** 运行在端口 8080，只处理 REST API（文件上传、角色管理、健康检查）。不代理 WebSocket 流量——Engine 直接处理 WebSocket 以减少触摸反馈延迟。

### Engine → Eye App（视频控制）

```json
{
  "type": "play_emotion",
  "video_path": "/characters/trump/happy.mp4",
  "transition": "crossfade",
  "transition_ms": 800
}
```

```json
{
  "type": "touch_feedback",
  "zone": "head",
  "channel": 2,
  "gesture": "tap",
  "effect": "ripple"
}
```

Eye App 将区域映射到屏幕位置做叠加特效：
- `head` → 屏幕中心
- `body` → 屏幕边缘
- `left_leg` → 屏幕左下
- `right_leg` → 屏幕右下

### Engine → 手机（状态同步，每 500ms 推送）

```json
{
  "type": "state_sync",
  "emotion": "excited",
  "intensity": 0.85,
  "gait_bpm": 120,
  "legs": [0.9, 0.3],
  "touch_active": true,
  "character": "trump"
}
```

**字段说明：**
- `intensity`（0.0 - 1.0）：全局表演强度。同时缩放两样东西：(1) 充气占空比（步伐力度），(2) 步态速度范围。0.0 几乎不动，1.0 最大戏精模式。默认 0.7。
- `legs` 数组：每条腿当前充气比例。索引 0 = 左腿，索引 1 = 右腿。0.0 = 放气，1.0 = 最大充气。手机端只读显示用。
- `gait_bpm`：当前每分钟步数。
- `touch_active`：是否有触摸通道正在被激活。

### 手机 → Engine（用户控制指令）

```json
{ "type": "set_emotion", "emotion": "happy" }
{ "type": "set_intensity", "value": 0.6 }
{ "type": "set_gait_speed", "bpm": 60 }
{ "type": "switch_character", "name": "trump" }
{ "type": "list_characters" }
```

### 手机 → Server（文件上传，REST API）

```
POST   /api/characters                          # 创建新角色包（body: { name }）
GET    /api/characters                          # 列出所有角色包及完成状态
GET    /api/characters/{name}                   # 获取某个角色包详情（哪些情绪已有视频）
DELETE /api/characters/{name}                   # 删除角色包
POST   /api/characters/{name}/videos            # 上传视频（multipart: emotion=happy, file=happy.mp4）
GET    /api/characters/{name}/videos/{emotion}  # 获取视频文件（用于预览缩略图）
GET    /api/status                              # 系统健康状态
```

## 启动、关闭与异常处理

### 启动顺序

`scripts/start.sh` 按顺序启动所有进程（黑客松不需要 systemd）：

1. **FastAPI 服务器** 先启动（端口 8080）— 提供手机面板静态文件和 REST API
2. **Blob Engine** 其次启动（端口 8000）— 打开 WebSocket 服务，初始化 GPIO，开始主循环
3. **Eye App** 最后启动 — `chromium-browser --kiosk --disable-infobars --enable-features=VaapiVideoDecoder http://localhost:8080/eye`（全屏模式，硬件视频解码）

健康检查：`start.sh` 等每个进程绑定端口后才启动下一个。任何进程在 10 秒内启动失败则打印错误并退出。

### 关闭顺序

`scripts/stop.sh` 或在 `start.sh` 上按 Ctrl+C：

1. 发送 SIGTERM 给 Blob Engine → `atexit` 处理器将所有 GPIO 设为 LOW（气泵停、阀门关）→ 进程退出
2. 发送 SIGTERM 给 FastAPI 服务器 → 进程退出
3. 终止 Chromium 进程

**关键：崩溃时的 GPIO 清理。** Engine 同时注册 `atexit` 和 `signal.signal(SIGTERM/SIGINT)` 处理器调用 `GPIO.cleanup()`。确保即使意外终止，气泵停止、阀门关闭。

### 异常处理与降级

| 故障情况 | 系统行为 |
|---------|---------|
| Eye App WebSocket 断开 | Engine 继续运行。Eye App 每 2 秒自动重连。断开期间画面冻结在最后一帧。 |
| 手机 WebSocket 断开 | Engine 切回自动模式（触摸驱动）。手机显示「重新连接中...」并自动重试。 |
| MPR121 触摸传感器未检测到 | Engine 进入「无触摸」模式。仅手机控制可用。情绪保持 Calm 除非手机覆盖。 |
| 角色包不完整（不足 7 个视频） | 允许切换到该角色。缺失的情绪显示静态占位图（一个「?」脸）。 |
| 气泵连续运行超过 30 秒 | 看门狗自动切断气泵。冷却 2 秒后才可再次启动。 |
| Engine 进程崩溃 | GPIO 清理处理器触发。手动重新运行 `start.sh` 即可。 |

### Demo 模式

应对硬件在演示现场部分失灵的情况。通过手机 Settings 标签页或启动 Engine 时加 `--demo` 参数激活。

- **触摸传感器坏了？** → 手机面板成为唯一输入，情绪按钮正常使用
- **气动系统坏了？** → 屏幕表情视频照常切换，体验降级为「互动表情显示器」
- **屏幕坏了？** → 腿部仍然响应触摸走动，体验降级为「触摸响应走路机器人」

核心循环（触摸 → 情绪 → 输出）设计为每个输出通道（屏幕、腿部、手机）独立运行。任何组合出故障都不会导致 Engine 崩溃。

## 角色包格式

每个角色包是 `characters/` 下的一个目录，包含 `manifest.json`：

```json
{
  "name": "川普",
  "created_at": "2026-04-01T12:00:00Z",
  "videos": {
    "calm": "calm.mp4",
    "happy": "happy.mp4",
    "excited": "excited.mp4",
    "curious": "curious.mp4",
    "sleepy": "sleepy.mp4",
    "shy": "shy.mp4",
    "grumpy": "grumpy.mp4"
  }
}
```

`manifest.json` 在上传视频时由服务器自动生成。`videos` 字典只包含已上传的情绪——缺失的 key 表示该情绪视频尚未上传。

### 风险应对

- **硅胶腿固化失败：** 第一次浇注有气泡或漏气，没时间重来。备选方案：用乳胶手套/气球做简易软腿，绑在框架上。丑但能走。
- **Pi 视频播放卡顿：** 双视频交叉淡入淡出如果卡，降级为硬切换（无过渡）。用 480x480 分辨率。Chromium 启动参数加 `--enable-features=VaapiVideoDecoder --use-gl=egl` 开硬件加速。
- **屏幕太重走不了：** 降级到 1.28 寸 SPI 圆屏（10g），但需改用 Python framebuffer 播放器替代 Chromium。
- **演示现场硬件部分故障：** Demo 模式允许系统在任意硬件子集工作时正常运行。参见「异常处理与降级」章节。
